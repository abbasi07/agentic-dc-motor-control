1. PROJECT GOAL (REVISED — WHY AGENTS / LLMs ARE JUSTIFIED)
--------------------------------------------------------------------------------
NOT the goal: sort two fixed PID gain sets with an LLM (that is a for-loop).

YES the goal: an **adaptive control-design agent** that, under changing natural-
language specs and plant uncertainty, iteratively chooses *what to try next*
(structure + parameters), evaluates with deterministic tools, and produces a
grounded design rationale — something a static script cannot decide well when
the failure mode and next method are open-ended.

Research framing (attractive + defensible):
  "LLM-orchestrated adaptive controller design for uncertain plants (DC motor
   digital twin), with deterministic evaluation, constraint-aware redesign, a
   simulation certification gate, and ablation vs non-agent baselines —
   software/simulation only (no hardware)."

DELIVERY SCOPE LOCK: **strictly software / simulation**. No MCU, no motor
driver, no physical deploy. "Deployment" means exporting a certified controller
artifact + report from the digital twin, not flashing hardware.

Original diagram still maps, but roles are reinterpreted:

  NL Specs + System Dynamics
            |
            v
  LLM Orchestrator (adaptive strategy — NOT a dumb sorter)
      | interprets specs -> formal constraints + scenario priorities
      | decides WHICH specialist agent to call next (and why)
      | on FAIL: diagnoses scorecard -> redesign action (gains vs structure vs ID)
            |
            v
  Specialist design agents (tools)
      | PID / Tuner Agent
      | MPC Agent           (Optimize Trajectory)
      | Robust Agent        (Maximize Stability under mismatch)
      | RL / Adaptive Agent (Adaptive Learning)
      | Plant ID Agent      (optional: propose identification experiments in sim)
            |
            v
  Deterministic Simulation & Evaluation Node (code ONLY)
            |
            v
  Scorecard JSON (pass/fail + metrics) + Design Rationale (LLM, grounded)
            |
            v
  Simulation Certification Gate (release artifact if and only if tests pass)
            |
            v
  Export package: controller JSON/code + scorecard + rationale PDF/MD
  (NO hardware tool)

HARD RULE: Tools compute metrics and pass/fail. LLM never invents numbers from
plots. LLM plans, diagnoses, explains, and chooses next *action* under uncertainty.

2. WHAT FULLY JUSTIFIES LLM ORCHESTRATION (FEATURE PILLARS)
--------------------------------------------------------------------------------
These five pillars are REQUIRED for the research story. Ranking fixed candidates
alone does NOT justify an LLM.

P1 — Spec-to-Design Loop (NL -> formal requirements)
  Input: natural language performance/safety specs, e.g.
    "Settle under 1.2 s, overshoot < 8%, survive 0.05 N·m load after 1.5 s,
     keep |u| <= 12 V, prefer low effort if multiple pass."
  LLM output (structured JSON schema):
    - hard constraints
    - soft preferences / weights
    - scenario priorities (which tests matter most for THIS request)
  Code validates schema; rejects invalid / unsafe limits.
  Why LLM: specs are open-ended English; mapping to constraint sets is not a
  fixed enum early in the project and changes with user intent.

P2 — Uncertainty / Digital-Twin Stress Suite
  Plant is never assumed perfect. Always evaluate under:
    - nominal step
    - load disturbance
    - parameter mismatch (J, b, R scales)
    - measurement noise
    - (later) combined stress / worst-case batch
  LLM role: given which scenarios FAILED, decide whether to
    - retune same topology,
    - switch topology (PID -> robust / MPC),
    - or request plant identification / mismatch expansion.
  Why LLM: next structural choice depends on *failure pattern*, not one scalar.

P3 — Constraint-Aware Redesign (adaptive loop, not one-shot ranking)
  On fail, orchestrator must output a RedesignPlan, e.g.:
    - "tighten Ki / reduce Kd" (parameter tweak via tuner)
    - "switch to Robust agent — mismatch scenario failed"
    - "switch to MPC — saturation + constraint violation on voltage"
    - "run ID experiment — model trust too low"
  Loop: design -> evaluate -> diagnose -> redesign until pass or budget exhausted.
  Why LLM: branching redesign policy is combinatorial; fixed if/else becomes
  brittle as agents and failure modes grow. Ablation compares LLM policy vs
  hand-coded heuristic policy.

P4 — Simulation Certification / Release Gate (NOT hardware)
  System REFUSES to export a "certified" controller package unless:
    - all hard constraints pass on required scenarios, AND
    - (optional) worst-case / robustness batch OK.
  LLM role: explain BLOCK vs ALLOW using scorecard evidence; propose extra
  sim experiments or softer specs if blocked.
  Export artifact (SaaS-ready): gains/policy + scorecard JSON + rationale MD.
  Why LLM: human-facing gatekeeping + spec negotiation; code enforces the gate.
  Explicit non-goal: physical deployment / MCU flashing.

P5 — Ablation Study (scientific honesty)
  Always compare three pipelines on the SAME NL-spec test suite:
    A) Fixed script: enumerate controllers, evaluate, lexicographic/argmin
    B) Heuristic orchestrator: hand-coded if/else redesign rules (no LLM)
    C) LLM orchestrator: adaptive redesign + rationale
  Report: success rate, iterations to pass, cost (tokens/time), rationale quality.
  Why this matters: proves WHEN the agent helps vs when a for-loop wins.

3. LOCKED DESIGN DECISIONS
--------------------------------------------------------------------------------
- Control objective: angular SPEED (omega), not position (position = later lab).
- Plant fidelity: full armature-controlled DC motor (electrical + mechanical).
- Early-to-final labs: **simulation only** — no hardware deployment in scope.
- First controller family: PID; advanced topologies later, SAME interface.
- Environment: uv-managed virtualenv (.venv); Python >= 3.12.
- LLM: OpenAI API via openai + python-dotenv; key in .env (gitignored).
- Jupyter kernel: "Python (agentic-dc-motor)" (name: agentic-dc-motor).
- GitHub: origin -> https://github.com/abbasi07/agentic-dc-motor-control.git (main)
- Controller interface (ALL topologies):
    reset()
    step(measurement, reference, dt) -> u
- Evaluation is deterministic and shared (dc_motor.evaluate_controller).
- Scalar score in Lab_02 is a DEMO aggregate (not a universal standard); primary
  selection is constraints-first, then preferences from the Spec Agent.

Plant parameters (CTMS — University of Michigan DC Motor Speed example):
  J = 0.01 kg·m^2
  b = 0.1 N·m·s/rad
  K = 0.01  (Kt = Ke in SI)
  R = 1.0 ohm
  L = 0.5 H
  Source:
  https://ctms.engin.umich.edu/CTMS/index.php?example=MotorSpeed&section=SystemModeling

Plant equations:
  L * di/dt + R * i = V - Ke * omega
  J * d(omega)/dt + b * omega = Kt * i

Transfer function (speed from voltage):
  G(s) = omega(s)/V(s) = K / [(Js+b)(Ls+R) + K^2]

5B. SAAS PRODUCT FRAMING (SIMULATION-ONLY)
--------------------------------------------------------------------------------
Working product thesis:
  "Control Design Copilot" — paste plant model + NL performance specs; get a
  certified simulated controller package (parameters + scorecard + rationale)
  after an adaptive multi-method design loop under uncertainty.

Target audiences (priority order for an MVP):
  1) University labs / MOOC instructors / students  — teach feedback + AI tools
  2) Control/mechatronics engineers in SMEs         — accelerate early design
  3) Digital-twin / industrial software teams       — embed design API in PLM/CAE
  4) (Later) OEMs / robotics startups               — only after multi-plant library

Primary value propositions:
  - Time: hours of trial-and-error tuning -> minutes of guided design
  - Traceability: every claim tied to scorecard numbers + action log
  - Coverage: uncertainty scenarios by default (not one happy step plot)
  - Accessibility: NL specs for non-experts; rigorous export for experts
  - Comparability: PID vs MPC vs robust on one harness (vendor-neutral)

What makes it robust / versatile / useful (build these deliberately):
  ROBUST
    - Deterministic eval as source of truth (LLM cannot bypass)
    - Schema-validated DesignSpec; safety clamps on limits
    - Certification gate before export
    - Reproducible seeds, versioned plant/controller schemas
    - Ablation-proven when agent helps vs when script wins
  VERSATILE
    - Plant registry beyond one CTMS motor (templates: 1st-order, DC motor,
      simple servo; same Controller interface)
    - Pluggable design agents (PID/MPC/robust/RL) behind one API
    - Spec library + custom NL
    - Multi-tenant design jobs: async run, store artifacts
  USEFUL (SaaS UX)
    - Job API: POST /design {plant_id|params, nl_spec} -> job_id
    - GET /jobs/{id}: status, action_trace, scorecard, rationale, export.zip
    - UI: spec editor, live constraints preview, scenario dashboard, compare runs
    - Org features: saved plants, shared scorecards, audit log
    - Cost controls: max iterations, max tokens, cached evals

MVP SaaS slice (do not boil the ocean):
  DC motor digital twin + Spec Agent + PID tuner + LLM redesign loop +
  certification export + simple web/API. Add MPC/robust after retention is proven.

Non-goals for SaaS v1:
  - Hardware-in-the-loop, edge deploy, PLC codegen (can be v2+ narrative only)
  - Claiming "optimal control for any plant" without plant library + tests

Monetization sketches (for thesis/business case, not required to build now):
  - Education: course/lab seat license
  - Pro: per-design-job or seat + higher iteration budgets
  - Enterprise: private plant library + SSO + audit exports

Research <-> SaaS bridge:
  Research proves adaptive orchestration value (ablation).
  SaaS productizes the certified export + job API + multi-plant roadmap.

6. WHAT THE LLM IS ALLOWED / FORBIDDEN TO DO
--------------------------------------------------------------------------------
ALLOWED:
  - Parse NL specs -> DesignSpec JSON
  - Read scorecards / FailureDigest
  - Choose next tool/action under a budget
  - Write design rationale citing numeric evidence
  - Propose spec clarification questions if requirements conflict

FORBIDDEN:
  - Invent metrics or claim pass/fail without evaluate_controller output
  - Bypass the simulation certification gate
  - Claim hardware readiness (out of scope)
  - Directly emit free-form controller code that skips the interface/tests
    (controllers must be constructed via tools/agents and re-evaluated)
